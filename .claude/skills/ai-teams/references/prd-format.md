# PRD 文档格式规范

产品经理输出 `docs/prd.md` 时必须遵循以下格式：

---

```markdown
# [项目名称] - 产品需求文档

## 1. 项目概述

### 1.1 背景与目标
- 项目背景描述
- 核心目标（1-3 个）
- 成功指标

### 1.2 目标用户
- 用户画像描述
- 使用场景

### 1.3 技术约束
- 技术栈要求
- 部署环境
- 性能要求
- 兼容性要求

## 2. 功能需求

### 2.1 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|---------|--------|------|
| F-001 | xxx | P0 | xxx |
| F-002 | xxx | P1 | xxx |

### 2.2 功能详情

#### F-001: [功能名称]

**描述**：功能的详细描述

**用户故事**：作为[角色]，我希望[行为]，以便[价值]

**验收标准**：
- [ ] 标准 1
- [ ] 标准 2

**UI/交互说明**：（如适用）
- 页面布局描述
- 交互流程

（对每个功能重复以上结构）

## 3. 非功能需求

### 3.1 性能要求
- 响应时间
- 并发量
- 数据量

### 3.2 安全要求
- 认证方式
- 授权策略
- 数据保护

### 3.3 可用性要求
- 浏览器兼容
- 响应式设计
- 无障碍访问

## 4. 数据模型（初步）

### 4.1 核心实体
- 实体名称和关键属性
- 实体间关系

### 4.2 API 概要
- 主要 API 端点列表
- 请求/响应格式概要

### 4.3 模块间数据流概要
- 用户操作 → 前端组件 → API → 后端服务 → 数据库 的主要流转路径
- 标注关键的数据转换点

### 4.4 跨模块共享概念列表
- 列出前后端共享的核心概念（如用户、订单、文章等）
- 为每个概念标注推荐命名和数据格式
- 示例：用户 ID → userId, UUID v4 格式

## 5. 待确认事项

| 编号 | 问题描述 | 影响范围 | 默认假设 |
|------|---------|---------|---------|
| Q-001 | xxx | xxx | xxx |

## 6. 里程碑

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| MVP | 核心功能 | P0 |
| V1.1 | 增强功能 | P1 |
| V2.0 | 高级功能 | P2 |
```

## 撰写原则

1. **具体可量化**：避免"快速响应"，应写"API 响应时间 < 200ms"
2. **边界清晰**：明确说明"不包含"什么
3. **假设透明**：所有不确定的决策记录在待确认事项中
4. **优先级明确**：P0 = 必须有，P1 = 应该有，P2 = 可以有
5. **验收可测**：每个功能的验收标准必须是可以验证的
